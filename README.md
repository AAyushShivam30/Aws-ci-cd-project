# Aws-ci-cd-project
Project: Automated Java Application CI/CD Pipeline on AWSThis project demonstrates a functional Continuous Integration and Continuous Deployment (CI/CD) pipeline for a multi-tier Java application. The pipeline is designed to automate the lifecycle of the software from source code to a containerized image ready for the cloud.Project OverviewThe pipeline automates the building, testing, and deployment preparation for a Java application named vprofile-project. It uses Jenkins to orchestrate various tools that ensure code quality and reliable artifact storage.Technical StackSource Control: Git and GitHub.Build Tool: Apache Maven (using MAVEN3.9 and JDK17).CI/CD Tool: Jenkins (Declarative Pipeline).Code Analysis: SonarQube (with Sonar-Scanner 6.2) and Checkstyle.Artifact Management: Sonatype Nexus Repository Manager.Containerization: Docker.Cloud Platform: Amazon Web Services (AWS) using ECR (Elastic Container Registry).Notifications: Slack.Pipeline Stages1. Source Code ManagementThe pipeline pulls the latest code from the GitHub repository using specific branches such as 'docker' or 'atom' to ensure the correct version of the application is processed.2. Build and Unit TestingCompilation: The application is compiled using Maven, skipping tests initially to ensure a successful build.Testing: A dedicated 'UNIT TEST' stage runs mvn test to validate the code logic.Archiving: Successful builds result in the creation of .war artifacts, which are archived within Jenkins.3. Code Quality and SecurityCheckstyle: Performs static analysis to ensure the code adheres to defined coding standards.SonarQube Analysis: Scans the source code for bugs, vulnerabilities, and technical debt.Quality Gate: A safety mechanism that waits for SonarQube results. If the code does not meet the required quality threshold, the pipeline is automatically aborted.4. Artifact Upload (Nexus)The validated .war file is uploaded to a Nexus repository. This stage uses specific coordinates like GroupID 'QA' and dynamic versioning based on the build timestamp to maintain a history of releases.5. Dockerization and AWS IntegrationImage Build: The pipeline uses a Dockerfile (located in the project's multistage directory) to build a container image.AWS ECR Push: Jenkins authenticates with the AWS ECR registry using stored credentials. The built image is pushed to the repository with two tags: the specific Build Number and 'latest'.Cleanup: To manage disk space on the Jenkins server, old or dangling Docker images are removed after the push is complete.6. NotificationsThe pipeline includes a 'post' block that executes regardless of the build result. It sends a message to a Slack channel (#devopscicd) indicating whether the job succeeded or failed, providing a direct link to the build log.ConclusionThis project proves my ability to integrate multiple DevOps tools into a single automated workflow. It highlights skills in cloud-native container management, automated testing, and secure artifact handling.
